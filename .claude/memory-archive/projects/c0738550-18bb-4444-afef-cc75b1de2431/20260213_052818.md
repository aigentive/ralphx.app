### Task Cleanup Service Infrastructure Pattern (0072a965-f222-4ea1-a081-8ee84c44977f)

**Summary:** - Backend `cleanup_task` command provides full cleanup pipeline (agents, git, FK refs, DB) via `TaskCleanupService::cleanup_single_task()`
- Frontend mutation hook (`cleanupTaskMutation`) already wired in multiple surfaces (TaskCard, Graph, Detail panels)
- Context menu integration: existing mutations often not surfaced in right-click menus — requires explicit wiring in `TaskContextMenuItems`

## Context
The RalphX task cleanup pipeline is well-established but the surface area for invoking it is incomplete. The task removal feature request revealed that full cleanup infrastructure exists but isn't consistently exposed in UI context menus.

## Cleanup Pipeline Architecture

The backend `TaskCleanupService::cleanup_single_task()` executes in strict order:
1. **Agent state transition** — Gracefully stop running agents (state machine transition)
2. **Git cleanup** — Delete worktrees and branches for the task
3. **FK cleanup** — Clear task_proposals, artifacts, and other foreign key references
4. **Database deletion** — Remove task record
5. **Event emission** — Emit `task:deleted` event to notify subscribers

This is accessed via the `cleanup_task` Tauri command and wrapped by `tasksApi.cleanupTask(taskId)` on the frontend.

**Not wired:**

When adding a destructive action to context menus:

2. Implement in surface components (TaskCard, TaskGraphView) — typically 2-3 lines calling existing mutation
3. Render in `TaskContextMenuItems` with destructive styling (Trash2 icon, red text)
4. Add confirmation dialog for destructive operations

**Key principle:** Mutations are often implemented before being discoverable. Always check if the backend API + hook already exist before writing new code.

### Deferred Plan Branch Creation at Task Execution (10ba7a79-d51a-42e2-a9a5-57e0834f2408)

**Summary:** Plan git branches should be created lazily when the first task executes, not when proposals are applied. This ensures branches fork from current main, preventing staleness from delayed execution. Database records exist at proposal time; git branch creation is deferred to execution time via resolve_task_base_branch().

## Pattern: Deferred Initialization for Time-Sensitive Resources

### Problem
Plan git branches were created eagerly when proposals were applied to kanban. If task execution was delayed (e.g., user applies proposals in morning, executes evening), the branch would be forked from a stale main state from hours earlier. This defeats the purpose of plan branches—to provide consistent, up-to-date context for all tasks.

### Solution
Defer git branch creation until the first task in the plan transitions to Executing state:

1. **Proposal Time**: Create DB record for plan branch (name, status, etc.)
2. **Execution Time**: Create actual git branch via `resolve_task_base_branch()` when task state transitions to Executing
3. **Idempotent**: Lazy creation checks if branch exists before attempting creation

### Implementation Details

In `resolve_task_base_branch()` (side_effects.rs):
- If plan branch record is Active and branch doesn't exist in git → attempt to create it
- On failure to create → log warning but still return plan branch name (don't fall back to main)
- On success → return plan branch name for use as task base branch
- Race condition handling: if creation fails, double-check with `branch_exists()` before falling back

### Code Location
`src-tauri/src/domain/state_machine/transition_handler/side_effects.rs:resolve_task_base_branch()`

### Related Files
- `src-tauri/src/commands/ideation_commands/ideation_commands_apply.rs` — removed eager branch creation
- `src-tauri/src/domain/state_machine/transition_handler/side_effects.rs` — added lazy creation

### Why This Matters
Plan branches serve as consistent context roots. Deferring their creation to execution time ensures they capture the current project state, not stale state from proposal time. This is critical for multi-day workflows where users plan tasks but execute them later.

## Streaming Content Block Architecture

### Current Behavior
- Backend sends `agent:chunk` events with partial text AND separate `agent:tool_call` events with tool metadata
- Frontend receives events via Tauri listener in `useIntegratedChatEvents` hook
- All text fragments accumulate into single `streamingText` string in state
- Tool calls render separately in streaming footer, but text stays merged in single bubble
- Result: Text message + "Add file" tool call + continuation text all appear in one bubble, losing natural flow

### Root Cause
Streaming state uses `streamingText: string` instead of preserving event boundaries.

### Solution
Change streaming state to track content blocks array matching backend structure:

```typescript
  | { type: 'tool_call'; toolCall: ToolCall }
  | { type: 'task'; taskSubagent: TaskSubagentCard }
```

During streaming:
1. Text chunks append to current text block
2. Tool call event creates new tool_call block (triggers new message boundary)
3. Final render shows blocks in order, text flows naturally around tool cards
4. When streaming completes, convert blocks → single Message object for DB

### rusqlite bundled SQLite enforces FK by default (31a262a1-5723-4df8-9d01-cf767070538c)

**Summary:** rusqlite with `bundled` feature compiles SQLite with `SQLITE_DEFAULT_FOREIGN_KEYS=1`, making FK constraints always-enforced even without explicit `PRAGMA foreign_keys = ON`. This can cause unexpected constraint violations if FK definitions lack `ON DELETE` actions.

## Context
RalphX uses `rusqlite` with the `bundled` feature, which compiles a vendored SQLite binary. This SQLite is compiled with `SQLITE_DEFAULT_FOREIGN_KEYS=1`, meaning foreign key constraint enforcement is **always enabled** at the database level.

## Key Finding
Despite no `PRAGMA foreign_keys = ON` existing anywhere in the Rust codebase, FK constraints are still enforced. This means any FK definition without an explicit `ON DELETE` action (CASCADE, SET NULL, etc.) will cause constraint violations when the parent row is deleted.

## Impact
- FK references without `ON DELETE` clauses can silently block deletions
- The application sees `FOREIGN KEY constraint failed` even though the pragma isn't explicitly set
- Cleanup operations can fail unexpectedly if dependent records exist

## Example
```rust
// This FK definition causes constraint violations on parent delete:
CREATE TABLE task_proposals (
  id TEXT PRIMARY KEY,
  created_task_id TEXT REFERENCES tasks(id)  -- ❌ No ON DELETE clause
);

// Fix: Add ON DELETE SET NULL (since column is nullable):
CREATE TABLE task_proposals (
  id TEXT PRIMARY KEY,
  created_task_id TEXT REFERENCES tasks(id) ON DELETE SET NULL  -- ✅ Correct
);
```

## Mitigation
1. **Always specify `ON DELETE` actions** when defining FKs (CASCADE, SET NULL, RESTRICT, NO ACTION)
2. **During schema review**, identify any `REFERENCES` without `ON DELETE` and add appropriate actions
3. **Defense-in-depth**: Application-level cleanup can pre-clear these references before deleting parent rows

- `src-tauri/src/infrastructure/database/migrations/`: All migrations should follow this pattern
- `src-tauri/src/infrastructure/database/schema.rs`: Review all FK definitions

### Scheduler context propagation failures cause silent merge target resolution (4f36b667-711f-474f-8020-7808ea16f938)

**Summary:** When state transitions require external context (like merge targets or plan branches), incomplete scheduler chain setup causes silent failures rather than errors. The pattern: TaskSchedulerService must call `.with_task_scheduler()`, and secondary schedulers must call `.with_plan_branch_repo()` to propagate context through the chain.

## Problem

When plan merge tasks transition to `PendingMerge` status, if the scheduler context chain is incomplete, merge targets cannot be resolved. The failure is silent—no error, no log, just stuck tasks in `pending_merge` state indefinitely.

## Root Cause Pattern

Scheduler context (plan_branch_repo, task context) must propagate through the entire chain:

```rust
// MISSING STEP: Build scheduler first
let scheduler = TaskSchedulerService::build_transition_service()
  .with_task_scheduler(/* .. */)  // ← REQUIRED
  .with_plan_branch_repo(/* .. */); // ← REQUIRED

// SECONDARY SCHEDULERS: Must also propagate context
scheduler
  .with_task_scheduler(/* .. */)
  .with_plan_branch_repo(/* .. */); // ← MUST NOT SKIP
```

Missing `.with_plan_branch_repo()` on secondary schedulers → merge target resolution fails silently.

## Impact

Tasks stuck in `pending_merge` or `merge_incomplete` states indefinitely. No database errors, no logs—just hung tasks.

## Detection Pattern

When debugging silent merge failures:
1. Check if tasks are in `pending_merge` with no visible error
2. Audit the scheduler chain setup in `TaskSchedulerService::build_transition_service()`
3. Verify all secondary schedulers have `.with_plan_branch_repo()` called
4. Verify all schedulers have `.with_task_scheduler()` called

- `src-tauri/src/application/scheduler_service.rs` (scheduler chain definition)
- `src-tauri/src/application/handlers/transition_handler.rs` (state transition logic)
- Any code building scheduler chains for multi-step transitions

### Task Git Branching: Local vs Worktree Mode & Merge Workflow (728b283c-afd3-4b33-87ba-f45458a7ab5b)

**Summary:** Two git modes enable single-task (Local) or parallel task (Worktree) execution. Task branches follow hierarchy: standalone tasks merge to main, plan tasks merge to plan feature branch which merges to main. Two-phase merge: programmatic fast path (rebase/merge or direct merge) then agent-assisted conflict resolution.

## Git Mode Comparison

| | Local | Worktree |
| --- | --- | --- |
| Enum | GitMode::Local | GitMode::Worktree |
| Isolation | Shared working directory | Separate directory per task |
| Parallelism | One task at a time (enforced) | Unlimited parallel tasks |
| Branch switch | git checkout on state entry | N/A (each worktree has own branch) |
| Dirty tree guard | Blocks Executing if uncommitted | N/A (isolated) |
| Agent CWD | project.working_directory | task.worktree_path |
| Cleanup on merge | Delete branch | Delete worktree + branch |
| Default | YES | — |
| DB fields | task.task_branch | task.task_branch + task.worktree_path |

**Config:** project.git_mode + project.base_branch (default: 'main') + project.worktree_parent_directory (default: ~/ralphx-worktrees)

**Local Mode Single-Task Enforcement:** TaskSchedulerService prevents more than one task entering Executing/ReExecuting/Reviewing/Merging states simultaneously. Scheduler blocks new tasks until slot frees.

## Branch Hierarchy

```
main (project.base_branch)
 ├─ ralphx/{slug}/plan-{artifact-id-8chars}     ← plan feature branch
 │   ├─ ralphx/{slug}/task-{task-id}            ← task branch (→ plan branch)
 │   └─ [merge task] plan branch → main         ← final plan merge
 └─ ralphx/{slug}/task-{task-id}                ← standalone task (→ main)
```

**Naming:**
- Task branch: ralphx/{project-slug}/task-{task-id}
- Plan branch: ralphx/{project-slug}/plan-{short-artifact-id-8chars}
- Worktree path: {parent}/{project-slug}/task-{task-id}
- slugify(): lowercase, non-alphanumeric → dash, trim edges

## Feature Branches (Plan-Level)

**Toggle:** project.use_feature_branches (default: true)
**Created at:** Plan apply (apply_proposals_to_kanban) or mid-plan (enable_feature_branch)

**On creation:**
1. Git branch ralphx/{slug}/plan-{id} from project.base_branch
2. DB record in plan_branches table (status: Active)
3. Auto-create merge task (status: Blocked, category: plan_merge)
4. Merge task blockedBy all plan tasks

**PlanBranch entity:** {id, plan_artifact_id, session_id, project_id, branch_name, source_branch, status, merge_task_id}

### Lazy Title Resolution for Async Agent Workflows (77cacc5e-8d9e-46e6-89dd-bd3ec338ab59)

**Summary:** When session titles are generated asynchronously (by session-namer agent) but needed immediately for git operations (plan merge commits), use lazy resolution at git-time instead of eager lookup at task-creation-time. This decouples agent scheduling from task creation and provides graceful fallbacks.

## Problem
RalphX's orchestrator-ideation creates tasks immediately, but session titles are generated asynchronously by a separate session-namer agent. When a user enables feature branch too quickly, the plan_merge task is created with `session.title = NULL`, so it falls back to a generic title ("Merge plan into main"). By the time the session-namer agent sets a real title hours later, the merge task already exists with the wrong title.

**Timeline of race condition:**
1. Orchestrator creates ideation session (title = NULL)
2. User enables feature branch → `enable_feature_branch()` called
3. Code tries to read `session.title` → None → falls back to "Merge plan into main"
4. Task created with wrong title
5. Session-namer agent runs later → sets real title
6. Merge commit uses task.title → too late, wrong value used

Evidence: Task created 2026-02-12 23:07:36, session updated 2026-02-13 03:07:09 (4h+ delta).

## Solution: Lazy Resolution at Git-Time
Instead of reading `session.title` when the merge task is created, resolve it at merge-commit-time (in `build_squash_commit_msg` side effect). This decouples:
- Session naming (async, can happen anytime)
- Task creation (happens immediately)
- Git operations (happen at merge, far in the future)

**Implementation pattern:**
```rust
// In side_effects.rs::build_squash_commit_msg, make async:
async fn build_squash_commit_msg(
    task_id: &TaskId,
    category: &str,
    ideation_session_id: Option<&IdeationSessionId>,
    session_repo: &dyn IdeationSessionRepository
) -> String {
    if category == "plan_merge" {
        // Fetch session at merge time (not task creation time)
        let session_title = if let Some(sid) = ideation_session_id {
            session_repo.get_by_id(sid).await
                .ok()
                .flatten()
                .and_then(|s| s.title)
                .filter(|t| !t.trim().is_empty())
        } else {
            None
        };
        
        // Use session title if available, otherwise fall back to task title
        let subject = session_title.as_deref().unwrap_or(task.title);
        format!("feat: {}", subject)
    } else {
        // Regular tasks: unchanged
        format!("feat: {}", title)
    }
}
```

## Why This Works
- **Decouples timing:** Session-namer can run whenever (async, lazy, parallel)
- **Graceful degradation:** If session title never gets set, uses task title as fallback
- **No race conditions:** Resolution happens when needed, not when created
- **Future-proof:** Applies to any session-derived metadata (plan name, tags, description) flowing into git messages

## Related Code
- `plan_branch_commands.rs:221-226` — tentative eager lookup (should revert)
- `side_effects.rs:~2713` — squash commit message builder (update to lazy)
- `IdeationSession::title` — `Option<String>` field
- `session-namer` agent — async title generation

## Trade-offs
- ✅ Decoupling wins over title accuracy timing
- ✅ DB lookup at merge-time is cheap (single row, already indexed)
- ❌ If session title is updated after merge task created but before merge, user sees updated title (acceptable: it's the latest state)

## Applicability
Use this pattern whenever:
- Async agents generate metadata (naming, classification, decisions)
- That metadata flows into git operations (commit messages, branch names)
- Task creation can't wait for agent completion

### Data Architecture: View-Driven Loading, Cache Hierarchy, Pagination (7ee50481-8d8a-4e65-9c3b-78cd1d28a433)

**Summary:** Universal patterns for server state: View-Driven (each view requests only what it needs), Query-First (server state → React Query, UI state → Zustand), Bounded Collections (never unbounded arrays, pagination required >100 items), Patch-First Cache (setQueryData > invalidateQueries > prefix invalidation), Separate Read/Write shapes. Pagination: cursor-based with CursorPage<T>, default 50/max 200, deterministic sort to prevent duplicates.

## 10 Core Data Patterns

| # | Pattern | Rule | Anti-Pattern |
| --- | --- | --- | --- |
| 1 | **View-Driven Loading** | Each view requests only what it needs. Ask: "What does THIS view need?" | Hydrating entire domain with nested relations for summary-only view |
| 2 | **Query-First Ownership** | Server state → React Query. UI state → Zustand. If from server, Query owns. | Syncing server data into Zustand and maintaining both manually |
| 3 | **Bounded Collections** | Never unbounded arrays. >100 items = pagination required from day one. | `get_all_X()` endpoints returning entire tables |
| 4 | **Patch-First Cache** | Event → patch cache from payload. Refetch only if event lacks data. Priority: `setQueryData` > targeted `invalidateQueries` > broad prefix. | Invalidating entire prefixes on every mutation (refetch storms) |
| 5 | **Separate Read/Write Shapes** | Writes return minimal confirmation. Reads return rich view-specific data. | Mutations returning "new state" conflicting with query cache |
| 6 | **Explicit Resource Budgets** | Every paginated endpoint: default size, max size, backend clamping. Define and enforce. | Trusting frontend to "be reasonable" with page size |
| 7 | **Migration Strategy** | **Pre-v1:** Ask user before feature flags (active development). **Post-v1:** new → behind flag → stable → remove old. | Big-bang replacements forcing all-or-nothing migrations |
| 8 | **Polling as Last Resort** | Prefer event-driven. Polling only for recovery fallback with strict TTL. If polling: add stop condition, backoff, max duration. | Multiple overlapping 2s polling that never stops |
| 9 | **Index Before Querying** | Add DB indexes for new query patterns BEFORE production. ORDER BY/WHERE on new columns = composite indexes. | Shipping pagination queries without indexes, N+1 in prod |
| 10 | **Measure Before and After** | Establish baseline BEFORE refactoring. Measure AFTER. Track: query count, payload bytes, render time, memory. | Claiming "faster" without metrics |

## Pagination Pattern

| Aspect | Specification |
| --- | --- |
| Backend | Cursor-based with `(cursor, limit, direction)`. Return `CursorPage<T>` with `items`, `next_cursor`, `has_more`. |
| Frontend | Use `useInfiniteQuery` for scroll-loading. Initial = latest page, older on demand. |
| Defaults | Page size: 50 default, 200 max. Clamp on backend. |
| Ordering | Deterministic sort (e.g., `created_at DESC, id DESC`) to prevent duplicates across pages. |

## Cache Invalidation Hierarchy

When event occurs (mutation, real-time update):
1. **First choice:** `setQueryData(key, payload)` if event contains full data → instant UI update, no refetch
2. **Second choice:** `invalidateQueries({ queryKey: ["specific", "key"] })` if event incomplete → targeted refetch
3. **Last resort:** `invalidateQueries({ queryKeyPrefix: ["prefix"] })` only for cascading updates

Avoid broad prefix invalidation unless multiple dependent queries need refresh.

### Data Architecture Patterns for Production Features (a3760793-22c7-4078-b866-ee7d27152582)

**Summary:** 10 universal patterns for query/state management extracted from ideation/chat refactor. Covers view-driven loading, query-first ownership, bounded collections, cache consistency, pagination, and measurement strategies. Applies to any feature with server state and real-time updates.

## Context

Extracted from the ideation/chat production-grade refactor plan (17-task import). These patterns transcend the specific chat context and apply universally to data-loading features.

## The 10 Patterns

### 1. View-Driven Data Loading
- Ask "What does THIS view need?" not "What data exists?"
- ❌ Hydrating entire domain objects when view needs summary
- ✅ Request minimal required fields

### 2. Query-First State Ownership  
- Server state → React Query cache (not Zustand)
- UI ephemeral state → Zustand/local store only
- Never sync server data in both places

### 3. Bounded Collections
- Never return unbounded arrays from APIs
- If collection can grow >100 items → pagination required from day one
- ❌ `get_all_X()` endpoints
- ✅ Cursor-based pagination with limits

### 4. Patch-First Cache Consistency
- Event occurs → patch query cache from event payload
- Refetch only if event lacks required data
- Priority: `setQueryData` > targeted `invalidateQueries` > broad prefix invalidation
- Prevents refetch storms

### 5. Separate Read/Write Shapes
- Mutations return minimal success confirmation
- Queries return rich view-specific data
- Don't overload mutations with full updated entities

### 6. Explicit Resource Budgets
- Every paginated endpoint: default size, max size, backend clamping
- Don't trust frontend to "be reasonable"
- Clamp on backend always

### 7. Polling as Last Resort
- Prefer event-driven updates
- Polling only for recovery fallback with strict TTL
- If adding polling: must have stop condition, backoff, max duration
- ❌ Multiple overlapping 2s intervals that never stop

### 8. Index Before Querying
- Add DB indexes for new query patterns BEFORE production traffic
- If query has `ORDER BY`/`WHERE` on new columns → add composite indexes first
- Prevent N+1 and table scans in production

### 9. Measure Before and After
- Establish performance baseline BEFORE refactoring
- Measure improvement AFTER (query count, payload bytes, render time, memory)
- Support claims with metrics

### 10. Pre-v1 Development Note
- No feature flags needed during active development
- Apply patterns directly without rollout mechanics
- Post-v1: pattern becomes "add new → migrate behind flag → stable period → remove old"

## Pagination Specification

**Backend:** Cursor-based with `(cursor, limit, direction)`. Return `CursorPage<T>` with `items`, `next_cursor`, `has_more`.  
**Frontend:** Use `useInfiniteQuery`. Initial load = latest page, older pages on demand.  
**Defaults:** 50 default, 200 max, clamp on backend.  
**Ordering:** Deterministic (e.g., `created_at DESC, id DESC`) to prevent duplicates across pages.  

## Cache Invalidation Hierarchy

1. `setQueryData` (patch from event) ← prefer
2. `invalidateQueries([specific, key])` ← targeted refetch
3. `invalidateQueries({ queryKey: [prefix] })` ← last resort

### Universal Data Architecture Patterns for Production Features (ab50292d-e853-4cee-88e7-470fac0d448b)

**Summary:** Extracted 10 reusable code quality patterns from the ideation/chat refactor that apply to any feature with server state and real-time updates: view-driven loading, query-first state ownership, bounded collections, patch-first cache consistency, read/write shape separation, resource budgets, polling fallback, database indexing, and performance measurement. Captured patterns avoid common mistakes like unbounded arrays, Zustand duplication, broad cache invalidation, and unmetered polling.

## Origin
Extracted from the Ideation + Chat Production-Grade Data-Fetch Refactor Plan during ideation session. Generalizes specific refactor patterns into universal principles applicable across all data-loading features.

## The 10 Patterns

### 1. View-Driven Data Loading
**Rule:** Each view requests only what it needs. Ask: "What does THIS view need?" not "What data exists?"
**Anti:** Hydrating entire domain objects with nested relations when view needs summary list only.
**Why:** Reduces payload size, improves cache hit rates, simplifies component logic.

### 2. Query-First State Ownership
**Rule:** Server state → React Query cache. UI state → Zustand/local store. If from server, Query owns it.
**Anti:** Syncing server data into Zustand and maintaining both manually.
**Why:** Single source of truth prevents desyncs, simplifies invalidation, reduces mutation complexity.

### 3. Bounded Collections
**Rule:** Never return unbounded arrays. If collection can grow >100 items, pagination required from day one.
**Anti:** `get_all_X()` endpoints returning entire tables.
**Why:** Prevents memory exhaustion, scales to production data, enables infinite scroll UX.

### 4. Patch-First Cache Consistency
**Rule:** Event occurs → patch query cache from payload. Refetch only if event lacks data. Priority: `setQueryData` > targeted `invalidateQueries` > broad prefix invalidation.
**Anti:** Invalidating entire query key prefixes on every mutation (refetch storms).
**Why:** Reduces network traffic, improves UI responsiveness, prevents waterfall refetches.

### 5. Separate Read/Write Shapes
**Rule:** Write commands return minimal confirmation. Read queries return rich view-specific data.
**Anti:** Mutation endpoints returning "new state" that conflicts with query cache logic.
**Why:** Decouples mutations from presentation layer, enables independent API evolution, reduces over-fetching.

### 6. Explicit Resource Budgets
**Rule:** Every paginated endpoint needs: default size, max size, backend clamping.
**Anti:** Trusting frontend to "be reasonable" with page size parameters.
**Why:** Prevents DOS-style query abuse, predictable server load, consistent performance.

### 7. Polling as Last Resort
**Rule:** Prefer event-driven updates. Polling only for recovery fallback with strict TTL. If adding polling: stop condition, backoff strategy, max duration.
**Anti:** Multiple overlapping 2s polling intervals that never stop.
**Why:** Reduces CPU/battery drain, lower server load, cleaner code than polling everywhere.

### 8. Index Before Querying
**Rule:** Add DB indexes for new query patterns BEFORE production traffic. If query has `ORDER BY`/`WHERE` on new columns, add composite indexes first.
**Anti:** Shipping pagination queries without indexes, discovering N+1/table scans in production.
**Why:** Prevents unplanned downtime, ensures pagination performs at scale.

### 9. Measurement Before and After
**Rule:** Establish performance baseline BEFORE refactoring. Measure improvement AFTER. Track: query count, payload bytes, render time, memory.
**Anti:** Claiming "it's faster" without metrics.
**Why:** Validates refactor ROI, provides data for future optimization, catches regressions.

### 10. No Fallback/Compat Periods in Pre-v1
**Rule:** Direct replacement. Remove old APIs inline as new ones roll out. No feature flags, no "sunset periods."
**Why:** Simplifies code, faster iteration, no technical debt carryover to v1 launch.

## Pagination Specification

- **Backend:** Cursor-based with `(cursor, limit, direction)`. Return `CursorPage<T>` with `items`, `next_cursor`, `has_more`.
- **Frontend:** `useInfiniteQuery` for scroll-loading. Latest page on load, older on demand.
- **Defaults:** 50 default, 200 max, clamped on backend.
- **Ordering:** Deterministic sort (e.g., `created_at DESC, id DESC`) to prevent duplicates across pages.

## Cache Invalidation Hierarchy

1. **setQueryData** (patch from event payload) ← Prefer
2. **invalidateQueries([specific, key])** (targeted refetch) ← Fallback
3. **invalidateQueries({ queryKey: [prefix] })** (broad invalidation) ← Last resort

## Application Matrix

| Scenario | Patterns |
|----------|----------|
| New data fetch | #1 (View-Driven), #3 (Bounded), #6 (Budgets) |
| Real-time feature (chat, notifications) | #4 (Patch-First), #7 (Polling Last Resort) |
| State refactor | #2 (Query-First), #5 (Read/Write Separation) |
| Performance issues | #9 (Measurement), #8 (Index), #3 (Bounded) |
| Large refactor | #10 (No Compat), #9 (Measurement) |

In pre-v1 development (no public users), avoid the overhead of:
- Feature flags for new APIs
- Backward-compatibility periods
- Dual implementations (old + new)
- Comparison testing (old vs. new)

Instead: build the right way once, delete old code immediately. Ship faster, accumulate zero technical debt.

### Plan Branch Lazy Creation Pattern (ac2648b9-b450-4956-b100-d0674ed5f19c)

**Summary:** Plan branch git branches are created lazily (on first task execution) rather than eagerly (at proposal time). DB record created at proposal time, git branch created when first task enters Executing state via resolve_task_base_branch(). Ensures branches fork from fresh main at execution time, avoiding stale branch issues.

## Problem
When plan branches were created eagerly at proposal application time, they would fork from whatever `main` existed at that moment. If tasks didn't execute immediately, the plan branch would be based on a stale version of `main`, causing integration issues later.

## Solution: Lazy Creation
DB record (plan_branches) created at proposal time, but actual git branch created lazily when first task enters `Executing` state.

### Timeline
1. **Proposal time** (`apply_proposals_to_kanban`): Create `plan_branches` DB record, don't create git branch yet
2. **First task execution** (`resolve_task_base_branch`): Create git branch from fresh `main`
3. **Merge** (`complete_merge_internal`): Delete branch & update plan_branches status

### Code Location
- **DB creation**: `src-tauri/src/commands/ideation_commands/ideation_commands_apply.rs:apply_proposals_to_kanban()`
- **Git branch creation**: `src-tauri/src/domain/state_machine/transition_handler/side_effects.rs:resolve_task_base_branch()`

### Lazy Creation Logic
```rust
// In resolve_task_base_branch():
if let Some(plan_branch) = plan_branch_repo.get_by_session_id(session_id)? {
  if plan_branch.status == PlanBranchStatus::Active {
    // Lazy: create branch if it doesn't exist
    if !GitService::branch_exists(&repo_path, &plan_branch.branch_name)? {
      if let Err(e) = GitService::create_feature_branch(&repo_path, &plan_branch.branch_name, &base_branch) {
        tracing::warn!("Failed to create plan branch: {}", e);
        // Still return plan branch name, don't fall back to default
        return Ok(plan_branch.branch_name);
      }
      tracing::info!("Created plan branch: {}", plan_branch.branch_name);
    }
    return Ok(plan_branch.branch_name);
  }
}
Ok(default)
```

### Race Condition Safety
If creation fails, second `branch_exists` check handles concurrent creation by another task:
```rust
if let Err(e) = GitService::create_feature_branch(...) {
  if GitService::branch_exists(...)? {
    // Another task created it, proceed
    return Ok(plan_branch.branch_name);
  }
  // Creation failed AND branch still doesn't exist, warn but continue
  tracing::warn!(...);
}
```

### Best-Effort Design
If lazy creation fails (e.g., no git repo in tests), the function still returns the plan branch name rather than falling back to default. This:
- Preserves test compatibility (mock repos won't have git)
- Lets downstream error handling deal with missing branches
- Avoids silently creating tasks from `main` when plan branch was intended

### Key Invariant
**Always return plan branch name when plan_branch.status == Active**, regardless of git creation success. The branch creation is an optimization; the DB record is the source of truth.

### Removed Code
Removed eager creation from `apply_proposals_to_kanban()`:
```rust
// REMOVED: No longer create git branch here
// if let Err(e) = GitService::create_feature_branch(...) { ... }
```

## Impact
- ✅ Plan branches always fork from fresh `main` at execution time
- ✅ No stale branch issues from delayed execution
- ✅ Backwards compatible: existing plan_branches records have `branch_exists()` check
- ✅ Race-condition safe: concurrent task creation handled by double-check
- ✅ Test-friendly: works with mock repos that have no real git

- Task base branch resolution uses `ideation_session_id` → `plan_branch_repo.get_by_session_id()`

### 3-Layer Tool Allowlist Verification Pattern (b1eeb481-f275-4b66-9148-8030c03ac0f8)

**Summary:** New MCP tools require explicit configuration across 3 independent layers: Rust spawn config (ralphx.yaml), MCP server TOOL_ALLOWLIST, and HTTP route mapping. ## 3-Layer Tool Allowlist Verification

The RalphX agent-MCP architecture enforces tool access through three sequential layers. A tool is only usable when ALL three layers grant access.

### Layer Architecture

**Layer 1: Rust Spawn Config** (ralphx.yaml mcp_tools array)
- Controls --allowedTools flag passed to Claude at spawn
- Missing here → tool not granted, Claude can't invoke

**Layer 2: MCP Server Filter** (ralphx-mcp-server/src/tools.ts TOOL_ALLOWLIST)
- Validates incoming tool calls match agent's allowlist  
- Missing here → 403 error returned to Claude

**Layer 3: HTTP Route Mapping** (ralphx-mcp-server/src/index.ts route handlers)
- Maps tool invocations to HTTP endpoints on Tauri backend
- Route mismatch → 404 error at runtime (common silent failure)

1. Tool name in ralphx.yaml mcp_tools array
2. Tool in TOOL_ALLOWLIST in tools.ts for that agent

4. End-to-end test to catch Layer 3 mismatches

- Simple ops: `/api/{operation}` (e.g., /api/create_child_session)

- Harder to verify coverage (invisible duplication)
- Maintenance burden: any changes must be applied in multiple places

### Lazy Plan Branch Creation Pattern (d28e641c-1976-427b-b4c0-e64f37ce6cf9)

**Summary:** Git feature branches for ideation plans are created lazily on first task execution, not at proposal time. Handles concurrent worktree execution via branch_exists checks.

# Lazy Plan Branch Creation

## Pattern Overview
Plan branches (ideation session feature branches) are **created on-demand** when the first task from the plan is executed, not when proposals are applied to the Kanban board.

## Why This Matters
- **Efficiency**: Branch creation deferred until actually needed (proposal → task workflow)
- **Concurrency Safety**: Multiple tasks from same plan may execute in parallel worktrees; idempotent design prevents failures
- **Workflow**: Proposals → Tasks → [first task execution triggers lazy branch creation] → merge flow

## Implementation Details

### Eager Creation (Removed)
Previously in `ideation_commands_apply.rs`:
```rust
let repo_path = PathBuf::from(&project.working_directory);
GitService::create_feature_branch(&repo_path, &branch_name, &base_branch)
```

### Lazy Creation (Added)
Now in `side_effects.rs::resolve_task_base_branch()` (lines 451-508):
```rust
if !GitService::branch_exists(repo_path, &pb.branch_name) {
    match GitService::create_feature_branch(...) {
        Ok(_) => { /* log success */ }
        Err(e) => {
            // Race condition: concurrent task may have created it
            if GitService::branch_exists(repo_path, &pb.branch_name) {
                // Safe to proceed - branch exists now
            } else {
                // Log and proceed anyway
            }
        }
    }
}
```

## Race Condition Handling
- When multiple tasks execute concurrently from same plan:
  1. First task to reach `resolve_task_base_branch` creates the branch
  2. Second task race: if `create_feature_branch` fails, check if branch exists via second `branch_exists` call
  3. If exists = concurrent task won; if not = actual error; both paths log and proceed

## Scope
Affects:
- `resolve_task_base_branch()` — determines base branch for a task (returns plan feature branch if available)
- `apply_proposals_to_kanban()` — no longer calls `GitService::create_feature_branch` directly
- Plan branch merge flow — target is now lazily-created branch

### Lazy plan branch creation at task execution (d899c187-1f2f-40ab-8ce1-6231e745a0e6)

**Summary:** Plan branch git branches are now created lazily when a task first executes (not when proposals are applied). This ensures branches fork from fresh `main` at execution time, not stale versions. Uses `resolve_task_base_branch()` as the deferral point.

## Problem
When proposals are applied to kanban, plan branches were created as git branches immediately. If tasks in that plan didn't execute for hours/days, the branch became stale—forked from an old `main` that had moved on.

## Solution
Defer actual git branch creation to task execution time:
1. DB record still created at proposal time (plan branch exists in schema)
2. Actual git branch created lazily in `resolve_task_base_branch()` when first task enters `Executing` state
3. This ensures branches fork from current `main`, not stale versions

## Implementation Pattern
- Check `branch_exists()` first (branch may have been pre-created or created by another task)
- Attempt `create_feature_branch()` if missing
- On failure, double-check `branch_exists()` for race conditions (concurrent task may have created it)
- If still doesn't exist, return plan branch name anyway (non-fatal)—downstream task branch creation handles missing bases

## Design Rationale
- **Best-effort creation**: Failures are non-fatal. Log warning but continue. Preserves test compatibility (mock repos).
- **Race-condition safe**: Double-check after create attempt handles concurrent execution
- **Graceful degradation**: If git operations unavailable (tests, CI), system still returns correct plan branch name—error handling deferred to downstream task operations

## Code Locations
- Deferral point: `src-tauri/src/domain/state_machine/transition_handler/side_effects.rs:resolve_task_base_branch()`
- Removed eager creation: `src-tauri/src/commands/ideation_commands/ideation_commands_apply.rs` (apply_proposals_to_kanban)

## Trade-offs
- Git state now split: DB branch records created immediately, git branches created later. Callers must handle non-existent branches gracefully.
- Race-condition window: Brief gap between plan branch record creation and git creation. Another task's concurrent execution may create the same branch.

### Spec deviation rationale: fail-forward error handling (ea1901c1-f9f9-4406-9b5f-61b6d85c7a16)

**Summary:** When spec prescribes silent fallback (return default) and implementation uses fail-forward (return failure indicator for downstream handling), the implementation is preferred for debuggability. Silent fallbacks mask configuration errors.

## Pattern: Fail-Forward vs. Silent Fallback Error Handling

### Context

Phase 150a spec for plan branch creation stated: "on failure, return default (project's base branch)"

Implementation instead: "on failure, return plan branch name and let downstream task execution catch the error"

### Analysis

**Silent fallback (spec's approach)**:
```rust
if create_feature_branch().is_err() {
    return default_branch  // e.g., "main"
}
```
Result: Task branches from `main` instead of plan branch. User sees success but unexpected branch context. Hard to debug why task branched from wrong base.

**Fail-forward (implementation's approach)**:
```rust
if create_feature_branch().is_err() {
    warn!("branch creation failed");
    return plan_branch_name  // Will fail downstream when task tries to branch from non-existent ref
}
```
Result: Task execution fails at worktree setup with clear error "branch not found". Configuration problem is visible and debuggable.

### Decision Framework

**Prefer fail-forward when**:
- Error indicates configuration/state problem (not transient failure)
- Downstream layer is equipped to handle and report the error
- Silent fallback would mask the root cause
- Error recovery requires user intervention (fixing plan setup)

**Use silent fallback only when**:
- Fallback is an intentional graceful degradation
- Fallback is equivalent in meaning (not just a default)
- User can infer the fallback from context

### Application in RalphX

Plan branch setup errors are configuration problems, not transient. Task execution layer (worktree setup) is already equipped to report missing branches clearly. Therefore, fail-forward is superior.

### Related Pattern

See also: Error handling in `resolve_task_base_branch()` double-check for git race conditions. Layered error detection (fail-forward + downstream catch) is more robust than single-point silent recovery.

### Identical branch detection prevents infinite merge retry loops (f9bffe71-bd4e-4019-ac31-f7e8f317e22a)

**Summary:** Use early-return pattern with branch diff detection to avoid creating commits when branches are already merged. Prevents MergeConflict state from getting stuck retrying empty commits.

## Pattern: Early Detection of No-Op Merges

When merging branches that are already identical, git commands succeed but create no new commits. This can cause:
- Empty commit detection failures
- Infinite retry loops in state machines
- Stuck MergeConflict tasks

## Solution Architecture

```rust
// Before any merge operation, check if branches are identical
if branches_have_same_content(base, target)? {
    // Return success immediately — no merge needed
    return Ok(MergeResult::Success { commit_sha });
}

// Then proceed with normal merge flow
let output = git_merge_command()?;
```

```rust
fn branches_have_same_content(

    base: &str,
    target: &str,
) -> AppResult<bool> {
    let output = git_diff_quiet(base, target)?;
    Ok(output.status.success()) // true = identical
}
```

**Applied in all 4 merge functions**:
1. `try_squash_merge()` (line 1611)
2. `try_squash_merge_in_worktree()` (line 1717)
3. `try_rebase_squash_merge()` (line 1828)
4. `try_rebase_squash_merge_in_worktree()` (line 1952)

## Why This Works

- **Prevents race conditions**: Detects if someone else merged the same code before merge operation
- **Exits state machine cleanly**: Returns success so tasks progress, not stuck retrying
- **Defensive**: Complements stdout/stderr checking for robustness
- **Fast**: `git diff --quiet` exits immediately on first diff

- **Defense-in-depth**: Combines pre-check with commit-stage stdout+stderr validation
- **State machine safety**: Ensures no "invalid" states (stuck retrying nothing)

### Streaming state timing: duplicate messages from refetch race condition (fa81f364-45cb-43db-a544-cc31e5b51a21)

**Summary:** Duplicate messages appear during streaming due to timing window: when `agent:message_created` fires, frontend clears streaming state and React Query invalidates cache. ## Root Cause
During streaming, two events can overlap:
1. `agent:message_created` → clears `streamingText`, `streamingToolCalls`
2. React Query cache invalidation for message list
3. Message list query refetches from DB
4. Fresh message arrives in stream
5. Both DB message and fresh stream data render briefly

## Current Flow
```
Stream active:
```

### Task Deletion Failure: Foreign Key Constraint on task_proposals (053d0a0f-15c5-4b6f-b125-64dfc19ededf)

**Summary:** Task cleanup service fails with FOREIGN KEY constraint when deleting tasks that have proposals referencing them. Three FK columns lack ON DELETE CASCADE: task_proposals.created_task_id, artifacts.task_id, ideation_sessions.seed_task_id. Worktree/branch cleanup warnings are cosmetic; the real blocker is task_proposals.created_task_id.

## Root Cause

When TaskCleanupService attempts `DELETE FROM tasks WHERE id = ?`, SQLite enforces foreign key constraints and rejects the delete if:

**Primary blocker:** `task_proposals.created_task_id` references the task without `ON DELETE CASCADE`

**Secondary blockers:** Two other columns also lack CASCADE:
- `artifacts.task_id` (also nullable)
- `ideation_sessions.seed_task_id` (also nullable)

## Example Error Sequence

Task `c592ef33-74c2-4149-aa6c-57767080a43b` ("Fix MCP route paths for session linking tools"):
1. Task created with ID
2. Proposal `644ca7a2-c659-4811-988e-586823235d4b` has `created_task_id = c592ef33`
3. Cleanup service runs: `DELETE FROM tasks WHERE id = 'c592ef33'`
4. SQLite enforces FK: Proposal still references this task!
5. Error: "FOREIGN KEY constraint failed"
6. Cleanup service logs: "Failed to delete task..." and retries in loop

## Cleanup Service Behavior

The service doesn't know about FK constraints—it just tries:
```rust
DELETE FROM tasks WHERE id = ?
```

Expects the database to cascade. But three FK columns don't cascade, so:
- First attempt fails
- Retry after delete verification → still fails
- Task persists, cannot be cleaned up

## Solutions

### Option 1: Database Migration (RECOMMENDED)
Add `ON DELETE SET NULL` to all three columns:
```sql
ALTER TABLE task_proposals DROP CONSTRAINT fk_created_task_id;
ALTER TABLE task_proposals ADD CONSTRAINT fk_created_task_id FOREIGN KEY (created_task_id) REFERENCES tasks(id) ON DELETE SET NULL;

-- Same for artifacts.task_id and ideation_sessions.seed_task_id
```

**Rationale:** All three columns are nullable, so SET NULL is semantically correct. When a task is deleted, proposals/artifacts/sessions referencing it get NULL, which is safer than CASCADE-deleting the proposal.

### Option 2: Application-Level Cleanup
In cleanup service, before deleting task:
```rust
db.execute("UPDATE task_proposals SET created_task_id = NULL WHERE created_task_id = ?", [task_id])?;
db.execute("UPDATE artifacts SET task_id = NULL WHERE task_id = ?", [task_id])?;
db.execute("UPDATE ideation_sessions SET seed_task_id = NULL WHERE seed_task_id = ?", [task_id])?;
db.execute("DELETE FROM tasks WHERE id = ?", [task_id])?;
```

**Trade-off:** More verbose, but avoids migration if DB layer changes later.

- `src-tauri/src/application/task_cleanup_service.rs` (cleanup logic)
- `src-tauri/src/infrastructure/sqlite/schema.rs` (FK definitions)
- `src-tauri/src/infrastructure/sqlite/migrations/` (where fix would go)

### Handler composition pattern for error/success paths (17b2530c-9260-4c14-b0ee-2cca6679bd3f)

**Summary:** Extract error/success handlers as separate functions (not traits) when handlers operate on shared state and may trigger side effects like spawning. Return `bool`/`enum` to communicate control flow decisions rather than executing side effects within handlers.

## Pattern Overview
When refactoring large match arms (Ok/Err) from async closures into separate functions:

### What Works
- **Separate handler functions** (not trait impls) — simpler for side effects
- **Explicit control flow signaling** — handler returns `bool`/`enum` indicating "should caller return?"
- **Side effects at handler level** — spawn tasks, schedule transitions, emit events inside handler
- **Async context preservation** — handlers can call `.await` and nested async functions

### What Doesn't Work
- **Handler traits** — awkward ownership for shared state (Arc<Mutex<...>>)
- **Implicit control flow** — caller tries to guess what handler did
- **Split responsibility** — handler spawns retry, caller does error flow (confusing)

## Example: Error Handler with Early Return Signal
```rust
// Returns true = "I handled recovery by spawning retry", caller should return
async fn handle_stream_error(
    error: AgentError,
    context: &SharedContext,
    // ... other params
) -> bool {
    match classify_agent_error(&error) {
        AppError::StaleSession => {
            if let Ok(new_session) = attempt_session_recovery(...).await {
                // Spawn retry with new session
                spawn_send_message_background(
                    /* with new session */
                ).await;
                return true;  // Caller should exit error arm
            }
            // Fall through to normal error handling if recovery failed
        }
        _ => {}
    }
    
    // Normal error path: fail run, emit event, transition task
    let _ = fail_agent_run(...).await;
    emit_error_event(...);
    let _ = transition_task_to_failed(...).await;
    
    false  // Caller continues normally
}
```

## Caller Pattern
```rust
Match result {
    Ok(message) => {
        handle_stream_success(&message, ...);  // Fire and forget
    }
    Err(e) => {
        if handle_stream_error(e, ...).await {
            return;  // Handler spawned recovery, exit
        }
        // Unreachable (normal error path also returns/ends)
    }
}
```

## Benefits
- Single responsibility — each handler owns its path (success/error)
- Clear control flow — return value signals early exit
- Testable — handlers take owned params, easier to mock
- Reusable — handler can be called from multiple contexts

## Trade-offs
- Return type must encode control flow intent (`bool` for simple, `enum` for complex)
- Handlers must avoid shared mutable state (pass params instead)
- Async context requires handlers to be `async` too

### Race Condition Handling in Concurrent Lazy Initialization (1b812989-2def-4af9-8abc-b166ac343ee7)

**Summary:** When multiple tasks execute concurrently and trigger lazy git branch creation, use check-then-check pattern: attempt creation, then re-verify existence before treating as failure. This handles the race where Task A checks (not exists) → Task B creates → Task A creates (now fails but branch exists).

## Problem: Race Condition in Lazy Initialization

### Scenario

Multiple tasks from the same plan execute concurrently:

1. **Task A**: Checks `branch_exists()` → returns false
2. **Task B**: Checks `branch_exists()` → returns false (Task A hasn't finished yet)
3. **Task A**: Calls `create_feature_branch()` → succeeds
4. **Task B**: Calls `create_feature_branch()` → **fails (branch now exists)**

Task B shouldn't fail when the branch is actually in the desired state.

### Solution: Check-Then-Check Pattern

```rust
if !GitService::branch_exists(repo_path, &pb.branch_name) {
    match GitService::create_feature_branch(
        repo_path,
        &pb.branch_name,
        &pb.source_branch,
    ) {
        Ok(_) => {
            // Task A path: creation succeeded
            tracing::info!("Created deferred plan branch");
        }
        Err(e) => {
            // Task B path: creation failed, but check if branch exists anyway
            if GitService::branch_exists(repo_path, &pb.branch_name) {
                // Another concurrent task created it — success!
                tracing::info!("Deferred plan branch created by concurrent task");
            } else {
                // Actual failure
                tracing::warn!("Failed to create deferred plan branch");
            }
        }
    }
}
```

### Key Points

1. **First check** prevents redundant creation attempts if already exists
2. **Creation attempt** proceeds even if another task might be creating simultaneously
3. **Error re-check** determines if failure is due to concurrency (benign) or actual error (warn)
4. **Fallback behavior**: Even if creation fails, task can proceed using project base branch

### Where This Pattern Applies

- Any lazy initialization triggered by concurrent state transitions
- Git operations that can race (branch creation, tag creation)
- Idempotent operations where "already done" is acceptable outcome

### Race-condition detection: double-check after failed create (333b99e1-133d-4a45-a2b7-b8e0609ae19f)

**Summary:** When git create operations fail, double-check existence before assuming failure. Concurrent tasks may have created the resource between the first check and the creation attempt.

## Pattern
Race-condition-safe git operations follow this order:
1. **First check**: Does resource exist?
2. **Create attempt**: Try to create if missing
3. **Second check**: If create failed, check again before fallback

## Code
```rust
if !GitService::branch_exists(&repo_path, &branch_name) {
    // Resource missing, attempt creation
    if let Err(e) = GitService::create_feature_branch(&repo_path, &branch_name, source_branch) {
        // Create failed—but was it because resource now exists (race) or real error?
        if !GitService::branch_exists(&repo_path, &branch_name) {
            // Still doesn't exist—real failure
            tracing::warn!("Create failed: {}", e);
        }
        // If it exists now, race-condition resolved itself—no action needed
    }
}
```

## When This Matters
- Concurrent task execution: Two tasks for the same plan may both attempt branch creation
- Worktree operations: Multiple worktrees may attempt creation simultaneously
- Cloud sync: Synced branch creation from another machine

## Trade-off
- **Cost**: Extra git command (third `branch_exists` check)
- **Benefit**: Correct behavior under concurrency. Without it, create failure is misinterpreted as resource missing, leading to incorrect fallbacks

## RalphX Context
Task execution is orchestrated per-task, so plan branch creation is serialized by task execution sequence. However, the double-check is still prudent because:
- User could manually create branches in background
- CI/CD systems may create branches concurrently
- Worktree creation across multiple agents

### SQLite table recreation pattern for FK migration (33a2db90-9f1d-48c4-b9a9-e75c6313a077)

**Summary:** SQLite doesn't support `ALTER TABLE ... ALTER COLUMN`, so FK fixes require the 4-step table recreation pattern: create temp table with new schema, copy data, drop old table, rename temp. This is the standard approach for modifying column constraints in SQLite.

## Problem
When an FK definition needs updating (e.g., adding `ON DELETE SET NULL`), SQLite provides no `ALTER TABLE ... ALTER COLUMN` syntax. You must recreate the table.

## Solution: 4-Step Pattern

```sql
-- Step 1: Create temp table with corrected FK definition
CREATE TABLE task_proposals_new (
  id TEXT PRIMARY KEY,
  created_task_id TEXT REFERENCES tasks(id) ON DELETE SET NULL,
  -- ... other columns
);

-- Step 2: Copy data from old table
INSERT INTO task_proposals_new 
SELECT * FROM task_proposals;

-- Step 3: Drop old table
DROP TABLE task_proposals;

-- Step 4: Rename temp to final name
ALTER TABLE task_proposals_new RENAME TO task_proposals;
```

## In RalphX Migrations
All migrations in `src-tauri/src/infrastructure/database/migrations/` follow this pattern when updating table constraints. The migration number (v32, v33, etc.) auto-increments, and the system applies all pending migrations on startup.

## Key Considerations
- **Indexes**: Temp table recreates them automatically if defined inline; recreate separately if needed
- **Triggers**: Must be recreated after the table rename
- **Data integrity**: All data is preserved (INSERT INTO ... SELECT *)
- **Atomicity**: Each migration runs in a single transaction

## Example in RalphX
When fixing `task_proposals.created_task_id` FK, the migration:
1. Creates `task_proposals_new` with `ON DELETE SET NULL`
2. Copies all existing task proposals
3. Drops old `task_proposals`
4. Renames `task_proposals_new` → `task_proposals`
5. Updates schema version

This approach is **reversible** only if the previous migration schema is preserved (e.g., v31 → v32 → v31 requires both directions in the migration file).

## Problem

In RalphX's multi-execution workflow with `commit-lock`, when:

2. Second execution amends the commit with additional staged changes
3. Files are read/modified between executions

Existing tests can be **silently deleted** from the modified file if the second execution reads a stale/intermediate state.

## Example from task_2d2e9150

**Previous execution:**
- Added new test `test_try_complete_stale_rebase_has_real_conflicts` to `src-tauri/src/application/git_service/mod.rs`
- Staged and committed

**Current execution:** 
- Re-read file (already modified by previous execution)
- Added same test again
- But in doing so, accidentally removed tests `test_branches_have_same_content_identical`, `test_branches_have_same_content_diverged`, and `test_squash_merge_identical_branches_returns_success`

**Root cause:**
- File state between executions: intermediate (previous commit + new changes in edit)
- When appending new test near EOF, if insertion point calculation was off, trailing code got replaced instead of appended
- Or: staged changes from previous execution weren't visible to current execution until commit

## Detection Strategy

1. **Always run full test suite** on modified files to catch deletions
2. **Check git diff** against main: if insertions < expected, tests were lost
3. **Compare line counts** before/after: new test (~50 lines) + removed tests (3×20 = ~60 lines) = net deletion visible in diff
4. **Use `git show :file` + `git diff HEAD` sequence** to detect staged vs working tree mismatches

## Prevention Pattern

**For test additions in multi-execution contexts:**

```rust
// WRONG: Just append (can overwrite if file state uncertain)
append_test(new_test);

// RIGHT: Restore + append pattern
// 1. Diff current against main to find deleted tests
let deleted = diff_against_main(current_file)
    .filter(|line| line.was_removed && line.is_test_fn)
    .collect();

// 2. Read original from main
let original_tests = read_from_main(file_path)
    .extract_test_module();

// 3. Restore deleted tests
let restored = restore_tests(original_tests, deleted);

// 4. Append new test after restored
append_to_end(restored, new_test);
```

## RalphX-Specific Mitigation

- **commit-lock** enforcement: Acquire lock before any test file modifications
- **Pre-commit validation**: Run `cargo test --lib [module]` to fail fast on missing tests
- **Staged state awareness**: When re-reading files, check `git status` to see what's staged vs working tree
- **Execution sequencing**: When appending tests, **always check git log** to see if a similar test was recently added on this branch

## Time Cost

- **Detection**: 5-10 min (failing tests + debugging)
- **Recovery**: 10-20 min (git history archaeology + restoration)
- **Prevention**: <2 min (automated check in pre-commit hook)

### Task cleanup retry loop on FK constraint failure (4b872de5-29a5-4d91-8a57-9d406fd2d314)

**Summary:** Task deletion failures due to FK constraints trigger retry loops that repeatedly fail with identical errors. The cleanup service doesn't pre-clear dependent references, causing cascading timeouts and orphaned worktrees/branches. Defense-in-depth: explicit pre-delete FK clearing required.

## Problem Pattern

When a task can't be deleted due to FK constraint failures (e.g., `task_proposals.created_task_id` references it), the cleanup service enters a retry loop:

```
2026-02-13T03:38:48.827190Z WARN task_cleanup_service: Failed to delete task during batch cleanup
error=Database error: FOREIGN KEY constraint failed
2026-02-13T03:38:49.029025Z WARN task_cleanup_service: Task reappeared after delete, retrying cleanup
2026-02-13T03:38:49.046852Z WARN task_cleanup_service: Failed to delete task during post-delete verification
error=Database error: FOREIGN KEY constraint failed
```

The retries fail identically because the underlying FK dependency isn't cleared between attempts.

## Cleanup Sequence Today

1. `DELETE FROM tasks WHERE id = ?` (fails: FK constraint)
2. Detects failure, retries with same bare DELETE
3. Same FK blocks it again
4. Gives up, leaves worktree/branch orphaned
5. User sees cascading errors in logs

## Root Cause

The cleanup service doesn't know about all FK dependencies. It naively tries `DELETE FROM tasks` and expects SQLite's CASCADE to handle everything. But FK references without `ON DELETE` clauses block the delete entirely, preventing any row deletion.

Specific blockers found:
- `task_proposals.created_task_id REFERENCES tasks(id)` (no ON DELETE)
- `artifacts.task_id REFERENCES tasks(id)` (no ON DELETE)

Both are nullable — the correct semantic is `ON DELETE SET NULL`.

## Fix: Two-Layer Approach

### Layer 1: Schema Migration v32
Add `ON DELETE SET NULL` to FK definitions. This prevents the constraint violation at DB level:

```sql
CREATE TABLE task_proposals (
  id TEXT PRIMARY KEY,
  created_task_id TEXT REFERENCES tasks(id) ON DELETE SET NULL  -- ✅ Fixed
);
```

### Layer 2: Pre-Delete Cleanup (Defense-in-Depth)
Before attempting `DELETE FROM tasks`, explicitly clear dependent references:

```rust
// In cleanup_service.rs
async fn cleanup_task(&self, task_id: &TaskId) -> Result<()> {
    // Pre-clear dependent references (defense-in-depth)
    self.db.clear_dependent_proposals(task_id).await?;
    self.db.clear_dependent_artifacts(task_id).await?;
    
    // Now delete the task — no FK constraint will block
    self.db.delete_task(task_id).await?
}
```

## Why Both Layers

1. **Schema migration** → fixes problem for all future deletions
2. **Pre-clear logic** → handles edge cases where migrations don't cascade (and future missing ON DELETE clauses)
3. **Reliability** → explicit is better than implicit; don't rely on CASCADE when you can verify cleanup

### Best-effort git operations in test-compatible code (5af0208f-b7cc-4df1-915e-a52a3dd48baf)

**Summary:** Git operations in state machine handlers must be non-fatal to support mock testing. When git fails (e.g., no real repo), return intended state regardless—let downstream error handling deal with missing branches.

## Issue
Test mocks use fake repos (`/tmp/test`). When `GitService::branch_exists()` or `create_feature_branch()` is called against these, they fail silently (no real git repo). Tests then fail because the code falls back to defaults instead of returning the plan branch.

## Pattern
When git operations must coexist with mocked tests:
1. **Don't fallback on git failure**. Instead, return the intended state (plan branch name, not default).
2. **Treat git failures as non-fatal**. Log at `warn` level; continue execution.
3. **Rely on downstream error handling**. If the branch truly doesn't exist, downstream `create_feature_branch()` will fail appropriately when called.

## Code Pattern
```rust
if !GitService::branch_exists(&plan_branch_path, &plan_branch_name) {
    if let Err(e) = GitService::create_feature_branch(&plan_branch_path, &plan_branch_name, main_branch) {
        tracing::warn!("Failed to create plan branch {}: {}", plan_branch_name, e);
        // Continue anyway—return plan_branch_name, not default
    }
    // Double-check race condition
    if !GitService::branch_exists(&plan_branch_path, &plan_branch_name) {
        tracing::warn!("Plan branch still does not exist after creation attempt");
        // Still return plan_branch_name
    }
}
return plan_branch_name; // Always return intended state
```

## Why This Works
- **Tests**: Mock repos fail silently on git ops. Code returns plan_branch_name anyway. Tests validate business logic, not git operations.
- **Production**: Real repos succeed. Branch is created lazily at execution time.
- **Hybrid scenarios**: Partial git availability (e.g., no worktree but git available) still works.

## Caveat
This pattern only works when downstream code gracefully handles non-existent branches. In RalphX, `TransitionHandler.execute_task()` calls `create_feature_branch()` with proper error propagation, so non-existent plan branches fail with clear errors rather than silent misses.

```rust
    format!("feat: {}", title)
}
```

- ✅ Decouples session naming from feature creation (async-safe)
- ✅ Uses latest session metadata (self-healing if updated later)
- ✅ Graceful fallback (works even if session title never set)

### Best-Effort Git Operations in Domain Logic (6ec5534c-6c96-4ca7-9bdc-090d855d351f)

**Summary:** When domain state transitions invoke git operations, use best-effort semantics: attempt creation/modification but return success if git operation fails, letting downstream error handling deal with missing branches. Don't cascade git failures into state machine failures.

## Discovery: Separating Domain from Infrastructure Concerns

### Pattern: Best-Effort + Graceful Degradation

In `resolve_task_base_branch()`, the code attempts to create a plan branch but does not fail the state transition if creation fails. Instead:

```rust
if !branch_exists && !creation_attempted {
    if let Err(_) = create_feature_branch(...) {

        // Still return the plan branch name
        // Downstream task branch creation will handle missing base

}
```

1. **Separation of Concerns**: Domain logic (resolving which base branch to use) is separate from infrastructure concerns (whether git branch exists)
2. **Test Compatibility**: Tests using mock repos (no real git) don't fail state transitions—they just log warnings
3. **Production Resilience**: If git operation fails for transient reasons (permission, network), the state machine doesn't stall; downstream consumers handle the missing branch appropriately
4. **Correct Failure Propagation**: Task branch creation via `GitService::create_branch()` will still fail if base doesn't exist—that's where the error should surface

Initial implementation attempted to fail gracefully by returning `default` (main) if branch creation failed. This was wrong because:
- Task would be created from main instead of plan branch (silent data loss)
- Test would silently work but with incorrect base branch
- Better: return plan branch name anyway; let downstream creation fail if branch is missing

### Streaming message duplication root cause: timing window between state clear and DB refetch (6faa1461-c347-4543-9c80-e7a41bd0699e)

**Summary:** • During agent streaming, `agent:message_created` event clears streaming state and invalidates React Query cache
• Timing window exists where both DB-persisted messages and incoming stream messages render simultaneously
• Fix: Coordinate React Query refetch with streaming state transition to prevent overlap
• Affects: `ChatMessageList`, `useIntegratedChatEvents` hook

## Problem

When an agent finishes streaming and fires the `agent:message_created` event, the streaming UI:
1. Clears the streaming footer state
2. Invalidates React Query cache to refetch DB messages
3. This creates a race condition where both old streaming messages and newly fetched DB messages briefly coexist in the DOM

Result: Users see duplicate text messages in the chat history while streaming is active.

## Root Cause

The transition from streaming state → DB state is not atomic. React Query refetch is asynchronous, so there's a window where:
- Streaming footer still renders (with accumulated text + tool calls)
- React Query simultaneously fetches the new persisted message from the database
- Both render, appearing as duplicates

## Solution

Ensure the React Query refetch completes BEFORE clearing the streaming footer, or use a synchronous state transition:

```tsx
// Option 1: Sequential refetch
await invalidateQueries({ queryKey: ['conversation_messages'] });
clearStreamingState(); // Only clear after refetch completes

// Option 2: Use optimistic update with React Query onSuccess
invalidateQueries({
  onSuccess: () => clearStreamingState()
});
```

Alternatively: Don't invalidate the cache at all — directly append the complete message to the client-side message list once streaming finishes.

### Fail-forward vs silent fallback in plan branch creation (7c9ac51c-30f9-47d4-821c-498ec902223e)

**Summary:** When deferred branch creation fails, proceeding with the intended branch name (then failing downstream) is more debuggable than silently falling back to default branch. Allows error pipeline to catch and report failures clearly rather than creating confusing task-branch mismatches.

## Pattern: Fail-Forward Error Handling

### Problem
When creating a plan branch fails during task execution setup, two strategies exist:
1. **Silent fallback**: Fall back to the project's default branch (e.g., `main`)
2. **Fail-forward**: Log warning, proceed with intended plan branch name, let downstream fail

### Why Fail-Forward Wins
- **Silent fallback creates mystery**: Task claims to branch from plan but actually branches from `main` — no visible failure signal
- **Fail-forward has clear failure**: When branch doesn't exist, the task's own git operations fail with explicit error messages caught by execution pipeline
- **Debuggability**: Operator sees the actual failure in logs rather than silent behavior change
- **Spec deviation is justified**: Plans may say "return default" but implementation that fails forward is better for UX

### Implementation Pattern (Rust)
```rust
let branch_name = if let Ok(_) = GitService::create_feature_branch(...) {
    trace!("Branch created");
    pb.branch_name.clone()
} else {
    // Race condition? Check again
    if GitService::branch_exists(&pb.branch_name) {
        trace!("Branch exists, proceeding")
        pb.branch_name.clone()
    } else {
        // Genuinely doesn't exist — log warning and proceed
        warn!("Branch creation failed, proceeding with {}", pb.branch_name);
        pb.branch_name.clone()
    }
}
```

The double-check after failure handles the race condition where another process created the branch between our failed attempt and the check.

### SQLite FK constraint fix pattern with 4-step table recreation (9cdcd9d2-79dc-4671-9cbb-6d263fddf5f9)

**Summary:** Use PRAGMA foreign_keys OFF/ON to temporarily disable FK checks during schema changes. Apply 4-step table recreation (rename → create new → copy data → drop old) for each affected table. Idempotent migrations prevent errors on re-execution.

## Problem
SQLite FK constraint violations block schema changes when dropping/recreating tables that contain FK columns. Direct table recreation fails because the table being recreated is still referenced by other FK constraints.

## Solution Pattern

### 1. Disable FK Checks
Wrap the entire migration in a transaction with PRAGMA:
```rust
TX(db)?.execute("PRAGMA foreign_keys = OFF", [])?;
// ... migration steps ...
TX(db)?.execute("PRAGMA foreign_keys = ON", [])?;
```

### 2. Apply 4-Step Table Recreation
For each table needing constraint changes (e.g., adding ON DELETE SET NULL):
```rust
// Step 1: Rename old table
TX(db)?.execute("ALTER TABLE task_proposals RENAME TO task_proposals_old", [])?;

// Step 2: Create new table with updated constraints
TX(db)?.execute(&["CREATE TABLE task_proposals (", 
  "  id TEXT PRIMARY KEY,",
  "  created_task_id TEXT,",
  "  FOREIGN KEY (created_task_id) REFERENCES tasks(id) ON DELETE SET NULL",
  ")"].join(" "), [])?;

// Step 3: Copy all data from old table
TX(db)?.execute("INSERT INTO task_proposals SELECT * FROM task_proposals_old", [])?;

// Step 4: Drop old table
TX(db)?.execute("DROP TABLE task_proposals_old", [])?;
```

### 3. Preserve Indexes
Recreate indexes after table recreation:
```rust
TX(db)?.execute("CREATE INDEX idx_task_proposals_created_task ON task_proposals(created_task_id)", [])?;
```

### 4. Handle Dependent Tables
When other tables reference the table being recreated (e.g., proposal_dependencies FK pointing to task_proposals), the PRAGMA OFF disables those constraint checks temporarily. Order migrations so dependent tables don't block the primary table recreation.

## Idempotency Pattern
Wrap ALTER TABLE steps in error-safe conditions:
```rust
if let Err(e) = TX(db)?.execute("ALTER TABLE ...", []) {
    if e.to_string().contains("no such table") {
        // Already migrated, continue
    } else {
        return Err(e);
    }
}
```

## Key Caveats
- FK pragmas only affect future FK violations; existing constraints are not checked retroactively
- Transactions must be explicit (TX macro) to ensure atomic operation
- Test each migration separately with 10+ test cases covering: data preservation, index existence, nullable columns, FK survival on dependent tables

```rust
// In cleanup service
```

4. **Integration points** - Call before deletion at three locations:
   - Single task cleanup (main path)
   - Batch cleanup (loop)

### Merger Agent Stale Rebase Workflow: Idempotent Conflict Detection (c68eaf90-3952-4c6d-b781-f541fb6e5f71)

**Summary:** When merger agent re-enters merging state (e.g., retry after incomplete merge), try_complete_stale_rebase() provides idempotent conflict detection. If rebase was already completed by prior attempt, AlreadyComplete returned immediately. Prevents re-triggering same resolution logic.

## Pattern: Idempotent Rebase Recovery

**Problem:** Merger agent exits mid-merge, then gets re-spawned (user clicks Retry or timeout triggers reschedule). Without idempotent recovery, attempting stale rebase again could:
- Re-run git rebase --continue unnecessarily
- Lose context of what was already resolved
- Create duplicate conflict detection

**Solution:** `try_complete_stale_rebase()` returns `AlreadyComplete` when:
- No `.git/REBASE_HEAD` file exists → Rebase never started or already finished
- No conflict markers in remaining files → Conflicts already resolved
- No unfinished rebase todo → Previous attempt completed successfully

## Retry Scenarios

| Scenario | First Attempt | Retry Attempt | Result |
| --- | --- | --- | --- |
| Auto-resolve succeeds | CompleteSuccess | AlreadyComplete (idempotent) | Merge proceeds |
| Real conflicts found | HasConflicts | Agent fixes + complete_merge | Explicit completion |
| Incomplete git state | HasConflicts → agent timeout | AlreadyComplete | Skip stale recovery, assess current state |

## Merger Agent Integration

**Entry point:** `merging` state (agent spawned)

1. Get merge target (source/target branches)
2. Get task context (conflict files, current state)
3. Call `try_complete_stale_rebase()` implicitly via git checks
4. **If AlreadyComplete:** Proceed to merge
5. **If HasConflicts:** Read conflict markers, resolve manually
6. **If CompleteSuccess:** Rebase finished, merge branches

## Key Insight

Idempotency prevents the agent from re-executing already-completed recovery steps. This is critical because:
- Agent sessions can timeout mid-merge
- Retries are automatic (user clicks Retry button)
- Without idempotency, retry could overwrite correct resolutions with stale logic

By returning `AlreadyComplete`, the system says: "Someone already handled this rebase, move forward."


### SQLite FK constraint fixing via table recreation with PRAGMA control (c7690250-55eb-4863-b466-63c5c33564f6)

**Summary:** SQLite FK columns lacking ON DELETE clauses require 4-step table recreation (backup→drop→create+FK→restore). Use `PRAGMA foreign_keys = OFF` before DDL, re-enable after to safely handle dependent FKs. Pattern: swap tables during migration, preserving all indexes and data.

## SQLite Foreign Key Constraint Fix Pattern

### Context
When SQLite tables have FK columns without `ON DELETE` clauses (e.g., `created_task_id` or `task_id`), attempts to delete referenced rows fail with `FOREIGN KEY constraint failed`. RalphX encountered this with `task_proposals` and `artifacts` tables, where deleting a task orphaned dependent records instead of cascading.

### Solution: 4-Step Table Recreation

1. **Backup original**: `CREATE TABLE table_name_backup AS SELECT * FROM table_name`
2. **Drop original**: `DROP TABLE table_name`
3. **Recreate with corrected FKs**: `CREATE TABLE table_name (... fk_col ON DELETE SET NULL ...)`
4. **Restore data**: `INSERT INTO table_name SELECT * FROM table_name_backup; DROP TABLE table_name_backup`

### PRAGMA foreign_keys Handling

**Critical**: Wrap entire migration in FK pragma control:
```sql
PRAGMA foreign_keys = OFF;
-- All DDL operations here
PRAGMA foreign_keys = ON;
```

**Why**: When FKs are enabled, SQLite validates FK references during DDL. Disabling allows table drops/recreates of referenced tables without constraint violations. Re-enabling after ensures constraints are enforced on new data.

### Index & Data Preservation

- **Indexes**: Recreate separately after table restore (not automatically dropped with table)
- **Data**: SELECT + INSERT preserves all rows during backup→restore cycle
- **Dependent FKs**: proposal_dependencies FK→task_proposals survives the recreate because foreign_keys pragma is OFF during DDL

### RalphX v32 Example

Fixed `task_proposals.created_task_id` (task deletion orphans proposals) and `artifacts.task_id` (task deletion orphans artifacts) by:
- Adding `ON DELETE SET NULL` to both FK columns
- Using backup→drop→recreate→restore pattern
- Leveraging `PRAGMA foreign_keys = OFF` to allow `proposal_dependencies` FK to survive table recreation

### When to Use

✅ Fixing FK columns lacking ON DELETE/CASCADE  
✅ Changing FK column constraint behavior (SET NULL → CASCADE)  
✅ Adding new FK constraints to existing tables  
❌ Adding columns (use ALTER TABLE ADD COLUMN instead)  
❌ Renaming columns (use simple recreation without data)

### Identical-branch pre-check pattern prevents merge infinite loops (cabe5446-cbea-4b77-aa8b-206fba1e0d06)

**Summary:** Detect identical branches before expensive merge operations using `git diff --quiet` to return early and avoid empty-commit errors that trigger infinite MergeConflict retries. Implement as defense-in-depth: early detection + dual-stream error checking (stdout + stderr) for robustness.

## Problem
When merging branches with identical content, git emits "nothing to commit" (sometimes to stdout, sometimes stderr), causing merge functions to fail unpredictably. Failed merges trigger retry loops in MergeConflict task handlers, leading to stuck tasks.

## Solution: Three-Layer Defense

### 1. Early Detection (Pre-check)
Before any merge operation, check if branches have identical content:
```rust
fn branches_have_same_content(&self, from_ref: &str, to_ref: &str) -> AppResult<bool> {
    let output = self.git(["diff", "--quiet", from_ref, to_ref])?
    Ok(output.stdout.is_empty())
}
```

In each merge function:
```rust
let from_ref = format!("refs/heads/{}", &self.feature_branch);
let to_ref = format!("refs/heads/{}", &base_branch);
if self.branches_have_same_content(&from_ref, &to_ref)? {
    return Ok(MergeResult::Success { commit_sha }); // Early return
}
```

### 2. Dual-Stream Checking (Defense-in-Depth)
In commit error blocks, check BOTH stdout and stderr:
```rust
let commit_error = stdout.contains("nothing to commit") || stderr.contains("nothing to commit");
if commit_error {
    return Ok(MergeResult::Success { commit_sha });
}
```

### 3. Improved Error Messages
Always include both streams for debuggability:
```rust
return Err(format!("Failed to {}: stdout={}, stderr={}", operation, stdout, stderr).into());
```

- `try_squash_merge()` — line 1611-1617 (pre-check) + line 1668 (dual-check)
- `try_squash_merge_in_worktree()` — line 1717-1723 + line 1776
- `try_rebase_squash_merge()` — line 1828-1834 + lines 1891-1892
- `try_rebase_squash_merge_in_worktree()` — line 1952-1958 + line 2054

## Why This Works
1. **Pre-check**: Avoids expensive git operations on already-merged branches
2. **Dual-stream**: Handles both git stdout/stderr output patterns across different environments
3. **Fail-safe**: Falls back to secondary detection if pre-check misses edge cases
4. **Non-blocking**: Early success return prevents task retry loops

## Trade-offs
- Adds `git diff --quiet` call (minimal overhead, bounded by branch divergence)
- Slightly more lines of code, but prevents exponential retry cost
- Safe to both pre-check and post-check (idempotent)

### SQLite FK constraint fix migration pattern (d2673749-1813-4561-9e08-1ebc51e4ec8d)

**Summary:** SQLite requires explicit foreign key pragma handling during table recreation. The v32 migration demonstrates a robust 4-step pattern: disable FKs, recreate table with ON DELETE SET NULL, re-enable FKs, and verify constraints. This pattern applies across multiple tables and preserves existing data and indexes.

## SQLite Foreign Key Constraint Fix Pattern

### Problem
SQLite tables with foreign key constraint violations (orphaned references) cannot be altered in-place. The database enforces FK constraints that may be violated by existing data.

### Solution
Implement a 4-step table recreation pattern:

1. **Disable FK checks**: `PRAGMA foreign_keys = OFF;`
2. **Recreate table**: Create new table with desired FK changes (e.g., `ON DELETE SET NULL`), copy data, drop old, rename new
3. **Re-enable FK checks**: `PRAGMA foreign_keys = ON;`
4. **Verify indexes**: Recreate any indexes that were lost during rename

### Key Implementation Details

- **FK pragma toggle scope**: Disabling FKs temporarily allows constraint-violating data to be migrated safely
- **Data preservation**: Use `INSERT INTO new_table SELECT * FROM old_table` to maintain existing records
- **Index reconstruction**: Indexes are dropped when parent table is recreated; must be explicitly recreated with original DDL
- **Multiple tables**: Pattern applies independently to each table (e.g., `task_proposals` and `artifacts` in v32)
- **ON DELETE SET NULL**: Null out FK columns instead of cascade-deleting to preserve task records when related records are deleted

### Code Structure
```rust
// Execute batch of SQL statements (auto-commits after completion)
execute_batch(sql_statements: Vec<String>) -> Result<(), AppError>

// Helper functions for clean separation:
fn disable_fks() -> Vec<String> { /* PRAGMA foreign_keys = OFF */ }
fn recreate_table(/* table spec */) -> Vec<String> { /* CREATE, INSERT, DROP, ALTER */ }
fn recreate_indexes() -> Vec<String> { /* CREATE INDEX */ }
fn enable_fks() -> Vec<String> { /* PRAGMA foreign_keys = ON */ }
```

### Testing Strategy
- Verify ON DELETE behavior with explicit FK cascades
- Test idempotency (migration runs without error multiple times)
- Validate all indexes exist post-migration via `PRAGMA index_list(table_name)`
- Confirm nullable FK columns are properly set to NULL where expected
- Verify data preservation across all non-FK columns

### Limitations & Caveats
- **No explicit transaction wrapping**: `execute_batch()` auto-commits; if runtime error occurs mid-batch, no rollback occurs (mitigated by comprehensive tests)
- **Performance**: Table recreation is expensive on large datasets; background migration recommended for production
- **SQLite-specific**: This pattern does not apply to PostgreSQL or MySQL (both support ALTER TABLE with FK changes)

### Foreign key schema patterns: identifying nullable references (e0a07d21-6e79-4343-99df-8ae052f56cff)

**Summary:** When adding ON DELETE actions to FK constraints, nullable columns should use SET NULL (not CASCADE). Identify nullable FKs by checking column definition and semantic intent: if orphaning parent's child is acceptable, use SET NULL; if child must be deleted, use CASCADE. Pattern avoids data loss during cleanup.

## Decision Tree for ON DELETE Actions

When a FK definition lacks an `ON DELETE` clause, determine the correct action:

### 1. Is the FK column nullable?

```sql
-- Nullable FK → suitable for SET NULL
CREATE TABLE proposals (
  id TEXT PRIMARY KEY,
  created_task_id TEXT REFERENCES tasks(id) ON DELETE SET NULL
  -- ✅ Proposal survives without task reference
);

-- Non-nullable FK → must use CASCADE or RESTRICT
CREATE TABLE steps (
  id TEXT PRIMARY KEY,
  task_id TEXT NOT NULL REFERENCES tasks(id) ON DELETE CASCADE
  -- ✅ Steps deleted when task deleted (required to maintain NOT NULL)
);
```

### 2. What's the semantic intent?

#### Pattern A: "Child inherits parent's lifecycle"
- Example: `task_steps.task_id` (steps belong to task, delete task = delete steps)
- Action: **CASCADE**
- SQL: `REFERENCES tasks(id) ON DELETE CASCADE`

#### Pattern B: "Child can exist without parent reference"
- Example: `task_proposals.created_task_id` (proposal is orphaned if task deleted, but proposal survives)
- Action: **SET NULL** (if nullable) or **no constraint** (if not a true FK)
- SQL: `REFERENCES tasks(id) ON DELETE SET NULL`

#### Pattern C: "Parent cannot be deleted if child references it"
- Example: Audit tables, immutable references
- Action: **RESTRICT** or **NO ACTION**
- SQL: `REFERENCES tasks(id) ON DELETE RESTRICT`

### 3. Code Review Checklist

When reviewing schema changes or migrations:

```
☐ Every REFERENCES clause has an ON DELETE action
☐ Nullable columns use SET NULL or no action (not CASCADE)
☐ Non-nullable columns use CASCADE (to maintain NOT NULL invariant)
☐ Audit-style references use RESTRICT (immutability)
☐ CASCADE chains don't create unintended deletion cascades (e.g., delete task → delete proposals → delete notes)
```

### RalphX-Specific Examples

| Table | FK Column | Target | Nullable | Action | Rationale |
|-------|-----------|--------|----------|--------|----------|
| `task_proposals` | `created_task_id` | `tasks(id)` | YES | SET NULL | Proposal survives orphaning |
| `artifacts` | `task_id` | `tasks(id)` | YES | SET NULL | Artifact survives orphaning |
| `task_steps` | `task_id` | `tasks(id)` | NO | CASCADE | Steps require parent task |
| `task_notes` | `task_id` | `tasks(id)` | NO | CASCADE | Notes require parent task |
| `ideation_sessions` | `seed_task_id` | `tasks(id)` | YES | SET NULL | Session survives task deletion |

## Migration Safety

When fixing existing schema (adding ON DELETE to FK columns):

1. **Use 4-step pattern** (SQLite doesn't support ALTER COLUMN):
   - CREATE TABLE_new with correct FK
   - INSERT INTO table_new SELECT * FROM table
   - DROP TABLE table
   - ALTER TABLE table_new RENAME TO table

2. **Test for edge cases**:
   - Deleting task with existing proposals/artifacts → should orphan, not fail
   - Cascade chains → verify no unintended deletions

### Branch Existence Check Pattern for Best-Effort Operations (ebf0f564-494f-47b5-89d4-ca7f99ca8e3c)

**Summary:** Double-check pattern for concurrent git operations: if creation fails, re-check existence before giving up. Handles race conditions where concurrent task created the branch between first check and creation attempt.

## Pattern
When implementing best-effort git branch creation in concurrent systems, use this pattern:

```rust
if !GitService::branch_exists(&repo, "feature-branch")? {
  match GitService::create_feature_branch(&repo, "feature-branch", &base) {
    Ok(_) => {
      tracing::info!("Created feature branch");
    }
    Err(e) => {
      // Check again: another task may have created it
      if GitService::branch_exists(&repo, "feature-branch")? {
        // Success via concurrent creation, proceed
        tracing::debug!("Feature branch created by concurrent task");
      } else {
        // Creation genuinely failed, log and continue gracefully
        tracing::warn!("Failed to create branch, proceeding anyway: {}", e);
      }
    }
  }
  // Continue using feature-branch regardless of creation outcome
}
```

## Why This Matters
1. **Concurrent safety**: In Local mode, multiple worktrees can't execute in parallel, but merges can race. In Worktree mode, truly unlimited concurrency.
2. **Graceful degradation**: If creation fails (repo doesn't exist, git error), still proceed with the branch name. Let downstream operations fail with more context.
3. **Test compatibility**: Mock repos have no real git, so creation always fails. Double-check lets function continue anyway.

## Applies To
- `resolve_task_base_branch()` for plan branch creation
- Any lazy-initialization of shared git state in concurrent task execution

## Related Patterns
- Best-effort initialization (don't fail hard on non-critical setup)
- Idempotent operations (safe to call multiple times)
- Source-of-truth precedence (DB record over git branch existence)

### Git "nothing to commit" detection requires stdout check (f4f354ae-bdeb-4ff5-88d0-14df5e3a730f)

**Summary:** Git outputs 'nothing to commit' to stdout when merge creates no changes, not just stderr. Merge functions must check both streams to detect already-merged branches correctly.

## Problem
When git merge operations yield no changes (identical branches), `git commit` outputs "nothing to commit" to stdout, not stderr. Code checking only stderr misses this case, causing infinite retry loops.

## Solution Pattern
For any git command integration checking for specific output strings:

```rust
if stdout.contains("nothing to commit") || stderr.contains("nothing to commit") {

}
```

## Why Both Streams Matter
- Git is inconsistent: sometimes outputs to stdout, sometimes stderr
- Status messages ("nothing to commit") → stdout
- Error conditions → stderr
- Exit codes don't always distinguish between "nothing to do" and actual failure

### Plan branch lazy creation with race condition safeguards (f7686de9-292b-4162-9ff6-5a2a6c553ba9)

**Summary:** Deferred plan branch creation from ideation_commands_apply to task execution eliminates premature branch creation. Implementation includes idempotent branch_exists() check and double-check on creation failure to handle git race conditions safely.

## Pattern: Lazy Plan Branch Creation with Race Condition Handling

### Discovery
During task execution refactor (phase 150a), plan branch creation was moved from ideation command apply (eager) to task execution's `resolve_task_base_branch()` (lazy).

### Implementation Details

**Location**: `src-tauri/src/domain/side_effects.rs::resolve_task_base_branch()`

**Flow**:
1. Check `GitService::branch_exists(pb.branch_name)` → skip if already exists (idempotent)
2. If missing, call `GitService::create_feature_branch()` from `pb.source_branch`
3. On creation failure, double-check `branch_exists()` to detect race conditions
4. If branch truly missing after failure, log warning and return `pb.branch_name` for downstream handling

### Race Condition Safety

**Why double-check?** Between steps 1 and 2, another concurrent process might create the branch. The double-check on failure detects this:
- If second check returns `true`: another process created it successfully → proceed
- If second check returns `false`: actual creation failure → propagate warning, let downstream task branching fail with clear error

**Rationale for "fail forward" design**: Returning the plan branch name (even if creation failed) ensures the downstream task execution catches the error clearly. Silently falling back to project default branch (`main`) could mask the misconfiguration, making bugs harder to diagnose.

### Test Coverage

Existing test `resolve_task_base_branch_returns_feature_branch_when_active` exercises the happy path via mock repository. The failure path (`branch_exists=false` after creation attempt) is covered incidentally because `MemoryPlanBranchRepository` mocks git operations.

Note: Static `GitService` method calls in tests operate as no-ops on non-repo working directories, making real git failure scenarios untestable without dependency injection refactoring.

### Related Code

- Removed from: `src-tauri/src/application/ideation_commands_apply.rs`
- Moved to: `src-tauri/src/domain/side_effects.rs::resolve_task_base_branch()`
- Git calls: `branch_exists()` = `git rev-parse --verify` (fast, ~1 call per task start)

### Performance

Single git command per task execution. No measurable overhead.

### Key Lesson

When deferring resource creation from eager to lazy evaluation:
1. Always make creation idempotent (check-before-create)
2. Handle race conditions between check and create steps
3. Choose "fail forward" (let downstream handle) over "fail silent" (fallback) to preserve debuggability

### Pre-check pattern validation: branches_have_same_content() with early-return optimization (f7e44d05-3f15-4bf4-b306-8d45d8e8288b)

**Summary:** The branches_have_same_content() helper (git diff --quiet) is the defense-in-depth layer preventing merge attempts on identical branches. Pattern includes both early-return check AND post-merge validation of stdout/stderr.

## Pattern Context

Merge functions in `src-tauri/src/application/git_service/mod.rs` implement a two-layer defense against "nothing to commit" scenarios:

1. **Early-return layer** (line 1668, 1776, 1891, 2054): Check with `branches_have_same_content()` before attempting merge
2. **Post-merge validation layer** (lines 1671-1677, etc): Check stdout/stderr for "nothing to commit" after merge

Both layers are necessary because git behavior varies by scenario.

## Helper Function Implementation

```rust
pub fn branches_have_same_content(
    source_branch: &str,
    target_branch: &str,
) -> Result<bool, String> {
    let output = Command::new("git")
        .arg("diff")
        .arg("--quiet")
        .arg(source_branch)
        .arg(target_branch)
        .status()?;
    
    // git diff --quiet returns 0 if no diff (identical)
    // and 1 if there are differences
    Ok(!output.status().success())
}
```

Wait, logic check: if `--quiet` returns 0 (success), branches are identical, so return `true`. If returns 1, branches differ, so return `false`.

A row in another table has a FK reference to the task being deleted, and the FK definition lacks an `ON DELETE` action.

1. **Identify which table blocks the delete:**

   - **Fix the FK definition** (add ON DELETE SET NULL or CASCADE)

### Test restoration playbook for accidental deletion recovery (c3eaf8df-bc14-40fa-a6a8-7ee859ab4d78)

**Summary:** Step-by-step procedure to detect and recover deleted tests after unintended file modifications. Applicable when test file shows fewer lines than expected or git diff shows net negative insertions. Includes git archaeology (git show, git diff), restoration strategy (cherry-pick from main), and verification (full test suite).

## Scenario

You've modified a test file (e.g., adding new tests) and discover that **existing tests were accidentally deleted**.

**Symptoms:**
- `git diff` shows insertions (new tests) but also deletions (old tests)  
- Running `cargo test --lib [module]` fails with "test not found"
- Line count decreased when it should have increased

## Detection (2 min)

```bash
# 1. Check what tests exist on main
cargo test --lib [module] --verbose 2>&1 | grep "test " | wc -l
# Note the count

# 2. Check current branch
cargo test --lib [module] --verbose 2>&1 | grep "test " | wc -l
# Compare counts

# 3. Show git diff stats
git diff main -- [file_path]
# Look for: if insertions < (new tests × 50), something was deleted

# 4. Show deletions explicitly
git diff main -- [file_path] | grep "^-" | head -20
```

## Identification (3 min)

```bash
# 1. Get deleted test names
git diff main -- [file_path] | grep "^-.*fn test_" | sed 's/^-//' 
# Example output:
#   fn test_branches_have_same_content_identical() {
#   fn test_squash_merge_identical_branches_returns_success() {

# 2. Verify they exist on main
git show main:[file_path] | grep -A 2 "fn test_branches_have_same_content_identical"
```

## Recovery (5-10 min)

### Option A: Full file restore from main + re-apply new changes

```bash
# 1. Save current file state (with new additions)
cp [file_path] /tmp/current_backup.rs

# 2. Restore original from main
git checkout main -- [file_path]

# 3. View new test content from backup
grep -A 50 "fn test_try_complete_stale_rebase_has_real_conflicts" /tmp/current_backup.rs

# 4. Manually append the new test(s) from backup to restored file
# (use text editor or surgical git edit)

# 5. Verify: wc -l should show restored_count + new_test_count
wc -l [file_path]
```

### Option B: Cherry-pick deleted tests from main + merge

```bash
# 1. Extract deleted test functions from main
git show main:[file_path] | \
  sed -n '/^fn test_branches_have_same_content/,/^}$/p' > /tmp/deleted_test.rs

# 2. Review and manually insert into current file at correct location
# (typically before the new test or as part of the test module)

# 3. Verify all tests are present
cargo test --lib [module] --verbose 2>&1 | grep "test \|test result"
```

### Option C: git show + edit (most surgical)

```bash
# 1. View main version side-by-side
git show main:[file_path] > /tmp/main_version.rs

# 2. Identify exact line ranges of deleted tests in main
grep -n "fn test_" /tmp/main_version.rs

# 3. Extract just those lines
sed -n '5778,5820p' /tmp/main_version.rs >> [file_path]
# (adjust line numbers based on step 2)

# 4. Verify and test
cargo test --lib [module]
```

## Verification (3-5 min)

```bash
# 1. Run full test suite on modified file
cargo test --lib [module] --timeout 10m 2>&1 | tail -5
# Expected: "test result: ok. N passed; 0 failed"

# 2. Count tests matches main + new tests
cargo test --lib [module] --verbose 2>&1 | grep "^test " | wc -l
# Should = main_count + new_test_count

# 3. Verify git diff is additive only (no deletions)
git diff main -- [file_path] | grep "^-" | head -5
# Should show very few or no deletions (maybe imports, not test functions)

# 4. Inspect diff one more time
git diff main -- [file_path] | grep -E "^[+-].*fn test_"
# All should be "+" (additions), no "-" (deletions)
```

## Prevention (Pre-commit hook)

Add to `.git/hooks/pre-commit` or CI:

```bash
#!/bin/bash
# Prevent test deletion during modifications

for file in $(git diff --cached --name-only | grep "\.rs$"); do
  # Count test functions in staged version
  staged_tests=$(git show :"`$file" | grep -c "^fn test_")
  
  # Count in main
  main_tests=$(git show main:"$file" 2>/dev/null | grep -c "^fn test_")
  
  if [[ $main_tests -gt 0 ]] && [[ $staged_tests -lt $main_tests ]]; then
    echo "ERROR: Test deletion detected in $file"
    echo "  Main: $main_tests tests, Staged: $staged_tests tests"
    exit 1
  fi
done
```

| Step | Time | Notes |

| Recovery (Option A) | 5-10 min | Fastest for 1-3 deleted tests |
| Verification | 3-5 min | Full test suite |
| **Total** | **13-20 min** | Faster with prevention hook |

### Commit Lock Protocol for Parallel Stream Coordination (d0949e1c-5ca5-4652-966a-9f8814c499f8)

**Summary:** Prevents simultaneous commit conflicts when multiple streams execute in parallel. Single atomic Bash command (check + wait + acquire + commit + release) using `.commit-lock` file in project root with stream name + ISO timestamp. Stale lock = same content >30s old. CRITICAL: All operations must be one tool call with absolute paths via git rev-parse --show-toplevel.

## Lock File

**Location:** `.commit-lock` in project root (NOT subdirectories)
**Format:** Single line: `<stream-name> <ISO-timestamp>`
Example: `features 2026-01-29T10:30:45`

## CRITICAL: Atomicity Requirement

**The check-acquire-commit-release sequence MUST be a SINGLE Bash tool call.**

❌ **WRONG — Race condition:**
```
Tool call 1: Check if lock exists → "No lock"
Tool call 2: Acquire lock and commit
↑ Another agent creates lock between calls!
```

✅ **CORRECT — Single atomic command:**
```
Tool call 1: Check + wait + acquire + commit + release (all in ONE)
```

**Why:** Between separate tool calls, another agent can create a lock. You’d then overwrite it, causing concurrent commits.

## CRITICAL: Absolute Path Requirement

**ALL lock operations and git commands MUST use absolute paths to the project root.**

Agents may work in subdirectories. To ensure protocol works:
1. Determine project root: `git rev-parse --show-toplevel`
2. Use absolute paths for lock file operations
3. Run git commands from project root or use `-C` flag

**Why:** If lock is created in `src-tauri/` instead of project root, other agents won’t see it.

## Complete Commit Operation

**ALL of this MUST be in ONE Bash tool call:**

```bash
PROJECT_ROOT="$(git rev-parse --show-toplevel)" && \
LOCK_FILE="$PROJECT_ROOT/.commit-lock" && \
STREAM_NAME="<your-stream>" && \
MAX_WAIT=30 && \
WAITED=0 && \
while [ -f "$LOCK_FILE" ]; do
  LOCK_CONTENT=$(cat "$LOCK_FILE" 2>/dev/null)
  echo "Commit locked by $LOCK_CONTENT. Waiting 3s..."
  sleep 3
  WAITED=$((WAITED + 3))
  if [ $WAITED -ge $MAX_WAIT ]; then
    echo "Lock wait timeout, removing stale lock"
    rm -f "$LOCK_FILE"
    break
  fi
done && \
echo "$STREAM_NAME $(date -u +%Y-%m-%dT%H:%M:%S)" > "$LOCK_FILE" && \
git -C "$PROJECT_ROOT" add <files> && \
git -C "$PROJECT_ROOT" commit -m "<message>" && \
rm -f "$LOCK_FILE"
```

## Stale Lock Detection

A lock is stale ONLY if:
1. Content (stream + timestamp) is **SAME** as when first observed, AND
2. Timestamp is older than **30 seconds**

If content changed → NEW lock from different stream (NOT stale).
Commit takes ~10-15s max. 30s gives generous buffer.

## Error Handling

If commit fails while holding lock:
1. Release lock anyway (command template handles this)
2. Log failure
3. Do not retry immediately (let other streams proceed)

**Template ensures cleanup with subshell:**
```bash
(
  # ... acquire lock ...
  git -C "$PROJECT_ROOT" add <files> && \
  git -C "$PROJECT_ROOT" commit -m "<message>"
  RESULT=$?
  rm -f "$LOCK_FILE"  # ALWAYS release
  exit $RESULT
)
```

## Rules Summary

1. **SINGLE TOOL CALL** — check + acquire + commit + release in ONE Bash command
2. **ALWAYS use absolute paths** to project root for lock and git operations
3. **ALWAYS acquire lock BEFORE git add**
4. **ALWAYS release lock after commit** (success or failure)
5. **NEVER force-delete** another stream’s active lock (unless stale)
6. **Stale = SAME content + >30s old** (not just old timestamp)
7. **Always read lock content** (lock may change hands while waiting)
